## 1. Contexte & motivation

Neuf sous-projets sont terminés. Le `StorageModule @Global` est éprouvé par 2 consommateurs (Projects pour les images de projet, Avatar Profile pour la photo de profil). **CV** est le **3ème consommateur de S3**, qui valide la généralisation du pattern d'upload S3 sur un cas légèrement différent : un binaire PDF (vs des images), avec une logique singleton (1 seul CV courant) et un endpoint download backend (vs URL publique directe pour les images).

Le sous-projet livre la fonctionnalité "CV téléchargeable" du portfolio : l'admin uploade un PDF, le visiteur peut le télécharger via un lien public.

## 2. Scope

### Inclus

- **Schéma Drizzle** `cv_file` (uuid + fileName + fileKey unique + fileSize + mimeType + uploadedAt + updatedAt + 1 index sur uploadedAt)
- **Migration Drizzle** `0005_*.sql` (autogénérée standard)
- **`CvModule`** : controller + service + 1 DTO réponse + tests
- **4 endpoints sous `/cv`** :
  - `POST /cv/upload` (admin, multipart, max 10MB, MIME strict `application/pdf`, **upsert singleton** sur key fixe `cv/cv.pdf`)
  - `GET /cv` (public) → métadonnées du CV courant ou `null` si aucun
  - `GET /cv/download` (public) → stream backend PDF avec `Content-Disposition: attachment; filename="<latest.fileName>"`
  - `DELETE /cv` (admin) → 204 ou 404 si aucun CV
- **Validation upload** : `ParseFilePipe` avec `MaxFileSizeValidator(10MB)` + `FileTypeValidator(/^application\/pdf$/)`. `errorHttpStatusCode: 422`.
- **`StreamableFile`** pour le download (NestJS idiomatic).
- **Bucket** : `'portfolio-storage'` (existant), key fixe `cv/cv.pdf`.
- **Wiring** : `CvModule` dans `AppModule`. `MulterModule.register({ limits: 10MB })` dans le module. Schéma dans le barrel.
- **Tests** : ~10 tests service via `createMockDb()` + mock `StorageService`.

### Explicitement exclus

- **Pas de versioning** : singleton, 1 row max en DB, S3 PutObject écrase.
- **Pas de pagination** : 1 row max.
- **Pas de throttle** : POST est admin-only, GETs sont read-only.
- **Pas de pre-signed URLs** : download via backend proxy direct (Content-Disposition contrôlé).
- **Pas de support multi-format** (`.docx`, `.odt`) — PDF only (calque Hono).
- **Pas de prévisualisation backend** : si le frontend veut un PDF viewer, il consomme `GET /cv/download` ou utilise un viewer client-side.
- **Pas de notification email à l'admin sur upload** — YAGNI.
- **Pas de redirection 302 vers S3** : proxy intentionnel pour gérer le `Content-Disposition` avec le filename d'origine de l'admin.
- **Pas de cache headers** sur `GET /cv` ou `GET /cv/download` — pas de précédent dans le projet, ajout transverse plus tard si besoin.
- **Pas de protection CSRF spécifique** au download — c'est juste un GET public, pas de risque CSRF.

## 3. Décisions clés (résumé)

| Q | Choix | Conséquence |
|---|---|---|
| Q1 — stratégie stockage | A : Singleton key fixe `cv/cv.pdf`, upsert | Calque Avatar Profile, simple, S3 PutObject idempotent |
| Q2 — endpoint download | A : Backend proxy + StreamableFile | Filename de l'admin préservé via Content-Disposition |
| (lock) — MIME | strict `application/pdf` | Calque Hono |
| (lock) — taille max | 10MB | Calque Hono |
| (lock) — endpoints | 4 (upload admin, GET metadata, GET download, DELETE admin) | Couvre le besoin |
| (lock) — throttle | aucun | Admin-only POST + read-only GETs, pas de risque abus |

## 6. Schéma DB

`src/database/schema/cv-files.ts` :

```typescript
import {
  pgTable,
  uuid,
  text,
  integer,
  timestamp,
  index,
} from 'drizzle-orm/pg-core';
import { sql } from 'drizzle-orm';

export const cvFiles = pgTable(
  'cv_file',
  {
    id: uuid('id').primaryKey().default(sql`gen_random_uuid()`),
    fileName: text('file_name').notNull(),
    fileKey: text('file_key').notNull().unique(),
    fileSize: integer('file_size').notNull(),
    mimeType: text('mime_type').notNull().default('application/pdf'),
    uploadedAt: timestamp('uploaded_at', { withTimezone: true })
      .notNull()
      .defaultNow(),
    updatedAt: timestamp('updated_at', { withTimezone: true })
      .notNull()
      .defaultNow(),
  },
  (t) => ({
    uploadedAtIdx: index('cv_file_uploaded_at_idx').on(t.uploadedAt),
  }),
);

export type CvFile = typeof cvFiles.$inferSelect;
export type NewCvFile = typeof cvFiles.$inferInsert;
```

### Adaptations vs Hono

- `id` : `uuid` natif (vs Hono `text` + `crypto.randomUUID()`)
- Timestamps : `withTimezone: true`
- Naming index : `cv_file_uploaded_at_idx`
- TS export `cvFiles` (pluriel), table SQL `cv_file` (singulier)

> **Note** : avec le pattern singleton, l'index `uploadedAt` est techniquement inutile (1 seule row, ORDER BY n'a pas de coût). On le garde pour parité Hono et au cas où un futur changement passe en versioning. Coût négligeable.

## 8. Service

`src/cv/cv.service.ts` :

```typescript
import {
  Inject,
  Injectable,
  Logger,
  NotFoundException,
} from '@nestjs/common';
import { desc, eq } from 'drizzle-orm';
import { DRIZZLE } from '../database/drizzle.constants';
import type { Database } from '../database/drizzle.types';
import { cvFiles, type CvFile } from '../database/schema/cv-files';
import { StorageService } from '../storage/storage.service';

@Injectable()
export class CvService {
  private readonly logger = new Logger(CvService.name);
  private static readonly BUCKET = 'portfolio-storage';
  private static readonly KEY = 'cv/cv.pdf';

  constructor(
    @Inject(DRIZZLE) private readonly db: Database,
    private readonly storage: StorageService,
  ) {}

  async upsert(file: Express.Multer.File): Promise<CvFile> {
    // Ordre : S3 upload → DB upsert.
    // PutObject est idempotent : écrase systématiquement la key.
    await this.storage.upload(
      CvService.BUCKET,
      CvService.KEY,
      file.buffer,
      file.mimetype,
    );

    const [existing] = await this.db.select().from(cvFiles).limit(1);

    if (existing) {
      const [updated] = await this.db
        .update(cvFiles)
        .set({
          fileName: file.originalname,
          fileSize: file.size,
          mimeType: file.mimetype,
          updatedAt: new Date(),
        })
        .where(eq(cvFiles.id, existing.id))
        .returning();
      return updated;
    }

    const [created] = await this.db
      .insert(cvFiles)
      .values({
        fileName: file.originalname,
        fileKey: CvService.KEY,
        fileSize: file.size,
        mimeType: file.mimetype,
      })
      .returning();
    return created;
  }

  async findLatestMetadata(): Promise<CvFile | null> {
    const [row] = await this.db
      .select()
      .from(cvFiles)
      .orderBy(desc(cvFiles.uploadedAt))
      .limit(1);
    return row ?? null;
  }

  async download(): Promise<{ buffer: Buffer; metadata: CvFile }> {
    const metadata = await this.findLatestMetadata();
    if (!metadata) throw new NotFoundException('No CV uploaded');
    const buffer = await this.storage.get(
      CvService.BUCKET,
      metadata.fileKey,
    );
    return { buffer, metadata };
  }

  async remove(): Promise<void> {
    const metadata = await this.findLatestMetadata();
    if (!metadata) throw new NotFoundException('No CV uploaded');
    // Ordre : DB delete → S3 delete (cohérence Projects.remove après fix 8cf3104)
    await this.db.delete(cvFiles).where(eq(cvFiles.id, metadata.id));
    await this.storage.delete(CvService.BUCKET, metadata.fileKey);
  }
}
```

### Notes

- **Ordre upsert** : S3 upload d'abord, puis DB upsert. Cohérent avec `uploadAvatar` et `uploadImage`. Si DB échoue, l'orphelin S3 sera écrasé au prochain upload (idempotent).
- **Ordre remove** : DB delete d'abord, puis S3. Cohérent avec `Projects.remove` après le fix `8cf3104`. Si S3 échoue, DB est déjà clean (pas de référence cassée).
- **`BUCKET`/`KEY` en constantes statiques privées** : pas de getter env, calque Projects/Avatar.
- **`findLatestMetadata` retourne `CvFile | null`** : utilisé par `upsert` pour décider INSERT vs UPDATE, par le controller `GET /cv` pour `null`-passthrough, et par `download`/`remove` qui throw `NotFoundException`.

## 9. Controller

`src/cv/cv.controller.ts` :

```typescript
import {
  Controller,
  Delete,
  FileTypeValidator,
  Get,
  HttpCode,
  HttpStatus,
  MaxFileSizeValidator,
  ParseFilePipe,
  Post,
  Res,
  StreamableFile,
  UploadedFile,
  UseGuards,
  UseInterceptors,
} from '@nestjs/common';
import { FileInterceptor } from '@nestjs/platform-express';
import {
  ApiBearerAuth,
  ApiBody,
  ApiConsumes,
  ApiOperation,
  ApiResponse,
  ApiTags,
} from '@nestjs/swagger';
import type { Response } from 'express';
import { JwtAuthGuard } from '../auth/jwt-auth.guard';
import { CvService } from './cv.service';
import { UploadCvResponseDto } from './dto/upload-cv-response.dto';

@ApiTags('CV')
@Controller('cv')
export class CvController {
  constructor(private readonly cv: CvService) {}

  @UseGuards(JwtAuthGuard)
  @Post('upload')
  @HttpCode(HttpStatus.CREATED)
  @ApiBearerAuth()
  @ApiConsumes('multipart/form-data')
  @ApiBody({
    schema: {
      type: 'object',
      properties: { file: { type: 'string', format: 'binary' } },
      required: ['file'],
    },
  })
  @ApiOperation({
    summary:
      'Upload/replace CV (admin, max 10MB, application/pdf only)',
  })
  @ApiResponse({ status: 201, type: UploadCvResponseDto })
  @ApiResponse({ status: 401, description: 'Unauthorized' })
  @ApiResponse({
    status: 422,
    description: 'File too large or unsupported MIME type',
  })
  @UseInterceptors(FileInterceptor('file'))
  upload(
    @UploadedFile(
      new ParseFilePipe({
        errorHttpStatusCode: HttpStatus.UNPROCESSABLE_ENTITY,
        validators: [
          new MaxFileSizeValidator({ maxSize: 10 * 1024 * 1024 }),
          new FileTypeValidator({ fileType: /^application\/pdf$/ }),
        ],
      }),
    )
    file: Express.Multer.File,
  ) {
    return this.cv.upsert(file);
  }

  @Get()
  @ApiOperation({
    summary: 'Get CV metadata (public, returns null if no CV)',
  })
  findOne() {
    return this.cv.findLatestMetadata();
  }

  @Get('download')
  @ApiOperation({
    summary:
      'Download the CV (public, attachment with original filename)',
  })
  @ApiResponse({ status: 200, description: 'PDF stream' })
  @ApiResponse({ status: 404, description: 'No CV uploaded' })
  async download(
    @Res({ passthrough: true }) res: Response,
  ): Promise<StreamableFile> {
    const { buffer, metadata } = await this.cv.download();
    res.set({
      'Content-Type': metadata.mimeType,
      'Content-Disposition': `attachment; filename="${metadata.fileName}"`,
      'Content-Length': buffer.length.toString(),
    });
    return new StreamableFile(buffer);
  }

  @UseGuards(JwtAuthGuard)
  @Delete()
  @HttpCode(HttpStatus.NO_CONTENT)
  @ApiBearerAuth()
  @ApiOperation({ summary: 'Delete the current CV (admin)' })
  @ApiResponse({ status: 404, description: 'No CV uploaded' })
  remove() {
    return this.cv.remove();
  }
}
```

### Notes

- **`StreamableFile`** : NestJS idiomatic pour servir un Buffer/Stream avec headers personnalisés.
- **`@Res({ passthrough: true })`** : permet de set les headers tout en laissant NestJS gérer la sérialisation (vs `@Res()` sans passthrough, qui demande de tout gérer manuellement).
- **Filename safety** : `latest.fileName` peut contenir des caractères spéciaux. La RFC 6266 sur `Content-Disposition` est laxiste — la majorité des navigateurs acceptent les guillemets droits autour de n'importe quelle string. Si un nom contient un `"` ou `\`, la valeur sera mal formée. Dans l'esprit YAGNI : ne pas escape pour l'instant ; si problème en prod, ajouter un sanitize (regex `[^\w. -]` → `_`).
- **Pas d'`:id` dans les routes** : pas de risque de conflit d'ordre.
